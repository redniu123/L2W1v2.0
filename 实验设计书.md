## 0. 文档说明

### 0.1 文档目的
本文档用于统一记录 L2W1 项目的实验目标、数据口径、模型矩阵、路由策略、预算协议、评测指标、执行顺序和结果归档规则，确保实验过程可复现、可对比、可追踪。

### 0.2 当前冻结主线
本项目围绕中文手写体识别 / OCR 纠错 / 路由增强识别展开，核心问题不是单独比较大模型榜单，而是在预算约束下验证路由与端云协同框架是否有效。

当前论文主线冻结为：
- GCR 是传统全局置信度基线；
- BAUR 是核心预算感知路由器；
- DAR 是带领域耦合项的消融路由器；
- SH-DA++ 是最终整体框架。

## 1. 研究背景

### 1.1 任务定义
本项目任务为：给定单行文本图像与 OCR 初始结果，输出最终修正文本。

研究对象聚焦三个专业领域：
- geology
- finance
- medicine

### 1.2 问题动机
纯 OCR 在复杂手写、低置信度、生僻字、边界漏字和专业术语场景下容易出错，而高性能 VLM 的调用成本较高，因此需要预算感知的路由机制决定何时升级到更强模型。

如果不加约束地调用 VLM，还会出现自由改写、幻觉生成、格式坍塌和过纠错等问题，因此系统必须同时具备受限 Prompt、严格回填和熔断保护。

### 1.3 论文目标
本文要证明的不是“哪个 VLM 总榜第一”，而是“路由 + 预算控制 + 端云协同”的方法在多领域中文手写 OCR 纠错中有效，并且可以在多种模型上成立。

## 2. 研究目标与假设

### 2.1 总目标
在固定预算约束下，通过 BAUR 路由器与 SH-DA++ 框架，提高中文手写 / OCR 场景中的最终识别质量，并控制调用率、时延与可靠性。

### 2.2 子目标
1. 比较 GCR、BAUR、DAR 三类路由策略在相同预算下的差异。
2. 比较“仅 BAUR 路由”与“完整 SH-DA++ 系统”的差异。
3. 在 2 个本地 VLM 和 2 个云端 VLM 上验证该结论是否仍成立。
4. 让所有正式结果都能以逐样本 JSONL 的形式长期保存，以支持后续错误分析、论文绘图与审稿回复。

### 2.3 核心假设
- H1：BAUR 在相同 Budget 下优于 GCR。
- H2：DAR 作为带领域耦合项的消融路由器，不一定优于 BAUR，因为领域分数与轻量风险估计可能发生语义倒挂。
- H3：完整 SH-DA++ 优于仅有 BAUR 的基础链路。
- H4：严格回填和受限 Prompt 能降低幻觉与过纠错风险。

### 2.4 成功判定标准
- 在正式预算点上，主指标稳定优于 GCR baseline。
- 正式在线预算实验要求 `Actual Call Rate = B ± 0.5%`。
- 主表结果必须来自统一评测脚本、冻结数据划分和可追溯的结果文件。
- 每个主实验都必须同时具备 summary 文件与 per-sample JSONL 文件。

## 4. 方法设计

### 4.1 系统总览
整体系统命名为 **SH-DA++**。

系统结构包括：
- Agent A：刚性 OCR 基线
- Router：预算感知路由模块
- Agent B：语义纠错模块
- Backfill / Circuit Breaker：抗幻觉、拒改和熔断保护模块

### 4.2 Agent A
正式实验中，Agent A 固定为基础 OCR 模块，用于输出初始文本和原始置信度信号。

### 4.3 三类路由器定义

#### GCR：Global Confidence Router
- 内部别名：router1
- 旧称：API-Conf
- 定义：基于 OCR 引擎输出的全局置信度构造风险分数。
- 定位：传统弱基线，用于说明单一全局置信度不足以稳定捕捉局部错误。

#### BAUR：Budget-Aware Uncertainty Router
- 内部别名：router2
- 旧称：BAUR
- 定义：基于轻量不确定性特征的预算感知路由器。
- 定位：核心路由组件，用于在预算约束下选择最值得升级到 Agent B 的样本。

#### DAR：Domain-Augmented Router
- 内部别名：router3
- 旧称：DA-Router
- 定义：在轻量风险特征基础上引入领域增强项的路由器。
- 定位：消融 / 对照路由器，用于检验直接引入领域耦合项是否真正带来增益。

### 4.4 最终方法定义
**SH-DA++ = BAUR + 领域软提示词 + 严格回填 + 熔断机制**

在这一定义里，领域知识主要放在 Agent B 侧的软提示词中，而不是强行耦合到轻量路由打分内部。

### 4.5 运行链路
1. Agent A 输出 `T_A` 与原始置信度相关信号。
2. Router 计算风险分数与是否升级的决策。
3. 若升级，则调用 Agent B 进行受限纠错。
4. Backfill 对输出执行路径约束、回填和拒改。
5. 熔断监控器根据 CVR 等信号进行保护性降级。

## 5. 模型矩阵

### 5.1 主文模型
主文固定为 **2 个本地 VLM + 2 个云端 VLM**，不再额外扩模，以控制实验复杂度并保持论文主线聚焦。

| 类别 | 模型 | 角色 |
|---|---|---|
| Local | Qwen3-VL-8B | 本地主力模型 |
| Local | InternVL3-8B | 本地对照模型 |
| Cloud | Gemini 3 Flash Preview | 云端高吞吐模型 |
| Cloud | Claude Sonnet 4.6 | 云端强能力模型 |

### 5.2 使用原则
主实验先固定一个代表性 Agent B 比较路由器，再固定最优 Router 比较 4 个模型，从而把“路由收益”和“模型能力差异”分开讲清楚。

### 5.3 环境说明
当前本地硬件环境为：
- GPU：2080Ti
- VRAM：22GB
- Python：3.10
- CUDA：12.6

本地 VLM 更适合低并发、异步排队或离线批处理，而不适合宣传 300 / 500 级真并发。

## 6. 预算协议

### 6.1 Budget 定义
本项目中的 Budget 统一定义为 **Max Call Rate**，即在给定数据集上，允许被路由到 VLM 的样本最大比例。

Budget 不是：
- token 上限
- 显存上限
- 单样本延迟预算

### 6.2 正式在线预算点
正式在线预算点固定为：
- `B = 0.10`
- `B = 0.20`
- `B = 0.30`

### 6.3 在线预算控制协议
在线预算默认采用 Online Control，而不是只使用静态阈值。

正式实验要求：
- `Actual Call Rate = B ± 0.5%`

在线预算实验用于给出论文正式主表结果，并报告真实调用率、稳定性、时延和系统行为。

### 6.4 离线预算回放协议（Offline Budget Replay Protocol）
为提高预算实验效率，并支持预算-性能曲线分析，本项目引入离线预算回放协议。

该协议允许先对目标数据执行一次 100% 全量升级推理与缓存，再基于已保存的逐样本结果离线重建多个预算点下的结果文件。

#### 6.4.1 目的
离线预算回放用于：
- 快速分析不同预算点下的质量曲线；
- 复用一次 full-call 结果，减少重复调用成本；
- 为错误分析、案例筛选、绘图和领域比较提供统一数据底座。

#### 6.4.2 回放步骤
1. 在固定模型、固定 Prompt、固定数据划分下，对目标集合执行一次 100% 调用并缓存结果。
2. 保存完整 `full_budget_results.jsonl`。
3. 按 `router_score` 从高到低排序。
4. 对预算点 `B ∈ {0.05, 0.10, 0.20, 0.30, 0.50, 1.00}`，取前 `B` 比例样本作为 upgrade 集合。
5. upgrade 集合使用缓存的升级后结果。
6. 非 upgrade 集合回退到 `ocr_text`。
7. 为每个预算点重新生成逐样本 JSONL 和 summary 文件。

#### 6.4.3 使用边界
离线预算回放适合用于趋势分析、案例分析和曲线绘制。

离线预算回放不能完全替代正式在线预算实验，因为真实 call rate 控制、窗口波动、吞吐、排队、熔断触发和在线稳定性仍需通过独立在线运行来观测。

## 7. Prompt、回填与可靠性

### 7.1 Prompt 规范
受限 Prompt 是系统的正式组成部分，不同 Prompt 版本必须记录为 `prompt_version`，若文本发生变化则必须升级版本号。

结果进入论文主表时，必须能回溯到对应 Prompt 版本。

### 7.2 严格回填
系统默认启用严格回填与全局拒改机制，用于约束 VLM 输出，降低幻觉与不必要的自由改写。

### 7.3 熔断机制
系统通过 Circuit Breaker 监控拒改 / 异常信号，在风险升高时执行保护性降级，避免错误被系统性放大。

### 7.4 VLM 专属错误桶
结果分析中应单独统计 VLM 引入的错误类型，如：
- hallucination
- format collapse
- over-correction
- false rejection

这些错误不应与普通 OCR 错误完全混在一起。

## 9. 评测设计

### 9.1 核心指标
正式实验统一输出以下指标，并由统一脚本计算：

| 指标 | 定义 |
|---|---|
| CER | 最终文本与 GT 的字符错误率 |
| Boundary Deletion Recall@B | 边界漏字样本中被升级召回的比例 |
| Substitution CER | 对齐后只统计 substitution 的错误率 |
| CVR | 拒改或违反约束的比例 |
| AER | 升级样本中被真正接受编辑的比例 |
| call rate | 实际 VLM 调用比例 |
| p95 latency | 端到端延迟的 p95 |
| token usage | 模型消耗量 |

### 9.2 评测要求
- 所有指标必须基于统一评测脚本计算。
- summary 文件与 per-sample JSONL 必须同时保存。
- 每个预算点都必须能独立回查案例与错误类型。

### 9.3 输出文件
每轮正式实验至少输出：
- `per_sample_results.jsonl`
- `metrics_summary.json`
- `summary.csv`
- `domain_breakdown.csv`
- `failure_cases.csv`
- `backfill_log.jsonl`

## 10. 实验矩阵

### 10.1 主实验 A：路由对比
固定一个代表性 Agent B，比对三类 Router。

| 实验编号 | 路由器 | 正式名称 | 定位 | Budget |
|---|---|---|---|---|
| M1 | GCR | Global Confidence Router | 传统全局置信度基线 | 0.10 / 0.20 / 0.30 |
| M2 | BAUR | Budget-Aware Uncertainty Router | 核心路由组件 | 0.10 / 0.20 / 0.30 |
| M3 | DAR | Domain-Augmented Router | 领域增强消融路由器 | 0.10 / 0.20 / 0.30 |

### 10.2 主实验 B：系统级方法
固定 BAUR 为路由核心，比较“仅 BAUR”与“完整 SH-DA++”。

| 实验编号 | 系统 | 说明 |
|---|---|---|
| M4 | BAUR-only | 仅核心路由链路 |
| M5 | SH-DA++ | BAUR + 领域软提示词 + 严格回填 + 熔断 |

### 10.3 主实验 C：跨模型验证
固定最优 Router / 系统设置，比较 4 个 VLM。

| 实验编号 | 模型 | 类别 |
|---|---|---|
| V1 | Qwen3-VL-8B | Local |
| V2 | InternVL3-8B | Local |

## 11. 执行顺序

### 11.1 冻结数据
先完成 `labels.jsonl` 检查、正式 split 固定和数据统计导出。

### 11.2 冻结主配置
再固定：
- Agent A
- 代表性 Agent B
- 主文模型矩阵
- Prompt 版本
- 正式 Budget 点

### 11.3 先跑路由，再跑模型
先完成 M1-M5，再进入 V1-V4，因为这样能先回答“方法有没有用”，再回答“该结论是否跨模型成立”。

### 11.4 同时保存在线与离线结果
所有在线预算实验在正式运行时必须落盘 JSONL。

所有离线预算回放也必须为每个预算点单独落盘 JSONL。

在线与离线结果都要保留，后续分析时不允许只看其中一类。

### 11.5 OnlineBudgetController 的 warmup 说明
当前在线预算控制器采用滑动窗口 warmup 机制：
- 前 `window_size = W` 个样本只记录、不更新阈值；
- warmup 期间使用 `lambda_init` 做固定决策；
- 因此，当 `n_samples < W` 时，结果只能用于 smoke test / 联通性验证，**不能直接作为预算达标证明**。

工程执行要求：
- 小样本 smoke test 主要验证链路是否可跑、是否能落盘；
- 预算是否达标，必须在 `n_samples >= W` 或显式缩小 `window_size` 的 debug 模式下再判断；
- 正式主线验收时，必须单独记录 warmup 配置，避免误解 `Actual Call Rate`。

## 12. 风险与应对

| 风险 | 表现 | 应对 |
|---|---|---|
| BAUR 提升不明显 | 相比 GCR 增益弱 | 回查特征设计、阈值搜索与错误桶 |
| DAR 意外更强 | 消融定位失效 | 检查领域项定义及特征耦合方式 |
| 医学词表被质疑 | 审稿人质疑资源权威性 | 明确其为当前子域匹配词表，而非权威知识本体 |
| 云端成本高 | 无法横扫过多模型 | 保持主文 2 local + 2 cloud |
| 本地吞吐不足 | 高并发不可行 | 本地走低并发或离线批处理 |
| 后续无法做案例分析 | 只有汇总数值，没有逐样本输出 | 强制保存全部 JSONL |

## 13. 归档规范

### 13.1 版本命名
- 数据版本：`data_vYYYYMMDD`
- 路由版本：`router_vX.Y`
- 模型版本：`model_vX.Y`
- Prompt 版本：`prompt_vX.Y`
- 实验编号：`M / V / A / S + 两位数字`
- 结果目录：`results/YYYYMMDD_runXX/`

### 13.2 可追溯性
进入论文主表、主图和主要结论的所有结果都必须回溯到具体：
- `run_id`
- 数据版本
- 模型版本
- 路由版本
- Prompt 版本

### 13.3 结果有效性判定
以下结果一律视为无效：
- 未记录配置的结果
- 未保存逐样本 JSONL 的结果
- 预算口径不一致的结果
- 服务异常但未单独标注的结果
- 临时改 Prompt 却未升级版本号的结果
- `n_samples < window_size` 的 warmup-only online 结果被误当作正式预算达标结果

---

### 13.4 GitHub 中转目录方案
为便于云服务器结果回传与本地复核，仓库内允许使用 `cloud_result_sync/` 作为可提交的结果中转目录。

执行原则：
- 正式实验原始输出仍保存在 `results/`；
- `results/` 继续作为本地/服务器运行目录，不直接整体提交；
- 需要共享给本地分析或协作排查的 run，可从 `results/` 导出到 `cloud_result_sync/<tag>/`；
- 导出后通过 GitHub 推送，本地再 pull 使用。

推荐做法：
- 使用 `scripts/export_cloud_results.py` 将指定 run 从 `results/` 复制到 `cloud_result_sync/`；
- 同步 `export_manifest.json`，记录本次导出的源目录；
- 仅导出需要复核的 run，避免把整批大结果都推送到仓库。

## 14. 当前冻结结论

当前项目的正式研究叙事如下：
- GCR 是传统全局置信度基线。
- BAUR 是核心路由组件。
- DAR 是领域增强消融路由器。
- SH-DA++ 是最终整体框架。
- 主文模型矩阵固定为 Qwen3-VL-8B、InternVL3-8B、Gemini 3 Flash Preview、Claude Sonnet 4.6。
- 正式在线预算点固定为 10%、20%、30%。
- 离线预算回放用于趋势分析与案例回查，但不能替代正式在线预算实验。
- 在线结果和离线结果都必须完整保存，且每个预算点都必须生成单独 JSONL 文件。
